# Lessons

- 当用户指出 stop-serve 还不够时，不能只实现逻辑拦截；必须同步检查 CLI / 表格输出是否给出醒目、可见、低歧义的提示，必要时使用红色 warning 强化风险表达。
- 当用户明确指出在线优先会浪费高成本配额时，不能继续沿用“rqdata 优先、bundle 回退”的默认思路；应优先设计“bundle 优先、仅补缺口或显式开关在线刷新”的低流量策略。
- 当用户强调“不要残次品、要考虑未来扩展性”时，不能只给最小修补方案；必须把缓存层、来源优先级、元数据、并发、观测性和迁移路径一起设计完整，再进入 HARD-GATE。
- 当用户明确要求走高自治、可失败可回滚的原版 autoresearch 路线时，不能擅自收缩成保守的配置搜索器；应保留源码直接迭代的主路径，只补最小必要护栏，并把 live/runtime 边界写清楚。
- 当用户明确给出长时间自治预算（例如“你可以跑 8 小时，明早验收”）时，不能只跑一轮有限 catalog 后就停止；必须把 wall-clock 预算转成多轮搜索/扩表/复盘闭环，除非提前找到明确 champion 或被资源/错误阻断。
- 当用户给出“最多运行 N 分钟”这类预算时，要把它理解成上限而不是到点强停目标；应优先跑完更高价值候选，只有逼近上限或资源冲突时才停。
- 当用户要“复制粘贴即可运行”的长时任务命令时，不能把核心逻辑塞进超长 heredoc；应优先落到仓库内可复用的临时脚本（例如 `skyeye/tmp/`）并把启动命令缩成一行。
