# Lessons

- 当用户指出 stop-serve 还不够时，不能只实现逻辑拦截；必须同步检查 CLI / 表格输出是否给出醒目、可见、低歧义的提示，必要时使用红色 warning 强化风险表达。
- 当用户明确指出在线优先会浪费高成本配额时，不能继续沿用“rqdata 优先、bundle 回退”的默认思路；应优先设计“bundle 优先、仅补缺口或显式开关在线刷新”的低流量策略。
- 当用户强调“不要残次品、要考虑未来扩展性”时，不能只给最小修补方案；必须把缓存层、来源优先级、元数据、并发、观测性和迁移路径一起设计完整，再进入 HARD-GATE。
- 当用户明确要求走高自治、可失败可回滚的原版 autoresearch 路线时，不能擅自收缩成保守的配置搜索器；应保留源码直接迭代的主路径，只补最小必要护栏，并把 live/runtime 边界写清楚。
- 当用户明确给出长时间自治预算（例如“你可以跑 8 小时，明早验收”）时，不能只跑一轮有限 catalog 后就停止；必须把 wall-clock 预算转成多轮搜索/扩表/复盘闭环，除非提前找到明确 champion 或被资源/错误阻断。
- 当用户给出“最多运行 N 分钟”这类预算时，要把它理解成上限而不是到点强停目标；应优先跑完更高价值候选，只有逼近上限或资源冲突时才停。
- 当用户要“复制粘贴即可运行”的长时任务命令时，不能把核心逻辑塞进超长 heredoc；应优先落到仓库内可复用的临时脚本（例如 `skyeye/tmp/`）并把启动命令缩成一行。
- 当用户指出 AX1 策略“纸墙”时，不能把 rule baseline 残留和默认 LGBM 主路径混在一起判断；必须先核对当前 profile/manifest，再优先处理主链路上的真实缺口，例如 style feature universe breadth 和组合层相关性利用。
- 当用户强调 AX1 还在研发阶段、可以为长期演进牺牲兼容性时，不要为了旧代码兼容保留混乱接口；应优先保证分层、架构和模块边界正确，旧代码/无效代码可删可重写，未实现好的模块显式 `NotImplemented`，不要放半实现进主链路。
- 当用户明确暂缓 AX1 live 产品化、Black-Litterman、个股 sleeve、autoresearch 时，不能继续把这些列为本轮交付范围；应保留清晰 `NotImplemented` 边界，优先修默认 LGBM/风险/执行/验证主链路里的真实质量问题。
- 当用户强调 AX1 尚未开始训练、可为长期架构让步时，不要为了兼容旧 profile、旧测试或旧接口做适配层；不合理代码应删除或重写，宁可显式 `NotImplemented`，也不要把残次/半成品实现接入默认主链路。
- 当用户指出 AX1 训练前不应区分 PoC/生产时，不能用“PoC 可跑”降低标准；只要影响标签正确性、数据 contract、训练效率或默认参数可信度，就必须先作为训练硬门槛收口，否则训练结果没有解释价值。
- 当用户要求 AX1 默认参数“存在的问题一步到位”时，不能把默认值调整说成“第一版/试探版”；默认 contract 要按 correctness-first 一次性给出可解释、可验证、少返工的目标值，研究候选只作为后续验证范围。
- 当用户要求 AX1 做清晰、彻底的 breaking refactor 时，旧实现如果不在新主链路使用就直接删除；不要保留 fallback、compat bridge、旧测试断言或“以后可能用”的半残留代码。
- 当用户给出交易员对 AX1 promotion gate 的默认值建议时，应把它固化成显式 contract test；例如 `min_oos_rows=100/300`、`max_cv=0.50/0.40`、`bootstrap_ci_crosses_zero` soft gate，以及 bootstrap 默认 80% 单侧 positive CI。
