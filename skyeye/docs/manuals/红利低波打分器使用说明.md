# 红利低波打分器使用说明

迁移说明：

- 当前主实现位于 `skyeye/dividend_scorer/`
- 当前示例策略位于 `skyeye/examples/`
- 当前诊断工具位于 `skyeye/tools/`
- 后续自研命令与说明均以 `skyeye/` 路径为准

从 0 到 1 推荐先读 [README.md](./README.md)。

本文档说明如何在 RQAlpha 里使用 `512890` 红利低波打分器，包括：

- 同步打分器所需缓存到 SQLite
- 用 CLI 直接查看指定日期评分
- 在策略中调用 `get_dividend_score()`
- 用 `skyeye/strategy_scorer.py` 做滚动窗口回测打分
- 用 RQAlpha 原生回测跑单区间并导出图片
- 用参数审计脚本检查“是否过拟合 / 是否退化”
- 画出 `512890` 走势与打分曲线的对比图

## 1. 能力概览

打分器目前有两种使用方式：

- CLI：`python -m skyeye.dividend_scorer.main`
- RQAlpha Mod：策略里直接调用 `get_dividend_score()`

仓库里已经有一份示例策略和一份诊断工具：

- [dividend_low_vol_score_strategy_history_aware.py](../../examples/dividend_low_vol_score_strategy_history_aware.py)
  历史感知交易策略：用分数分位的历史均值和趋势决定主仓位，并只在价格/溢价/RSI 明显过热时强化减仓
- [dividend_score_vs_price_plot.py](../../tools/dividend_score_vs_price_plot.py)
  诊断工具：只负责画图，对比 `512890` 走势与 `dividend_score`

后续做策略研发和迭代，建议同时保留两条验证路径：

- `skyeye/strategy_scorer.py`：看多窗口稳定性，避免策略只在单一时间段有效
- `rqalpha run --plot-save`：看某一个具体区间的真实净值、基准对比和图片输出

## 2. 环境准备

### 2.1 推荐用 `uv` 统一环境

建议统一用 `uv`，原因：

- 避免把依赖装到错误解释器
- 把 Python 版本和 `akshare` 版本锁到项目本地环境
- 后续执行 `python -m ...` 和 `rqalpha run ...` 都走同一个环境

当前项目已经提供可选依赖组：

- `dividend-scorer`：固定 `akshare==1.18.38`

首次初始化：

```bash
uv venv --python 3.13
uv sync --python 3.13 --extra dividend-scorer
```

后续统一使用：

```bash
uv run python -m skyeye.dividend_scorer.main --help
```

### 2.2 默认路径

- SQLite 缓存默认路径：`~/.rqalpha/dividend_scorer/cache.db`
- bundle 默认路径：`~/.rqalpha/bundle`

### 2.3 准备 RQAlpha bundle

如果你后续要跑策略或诊断工具，需要先准备 bundle：

```bash
uv run rqalpha download-bundle
```

## 3. 第一步：同步打分器缓存

如果你只想先把缓存同步好，不立刻输出评分，可以单独运行：

```bash
uv run python -m skyeye.dividend_scorer.main \
  --sync-only \
  --end-date "$(date +%F)"
```

同步说明：

- 默认执行 `uv run python -m skyeye.dividend_scorer.main` 时，也会先自动检查并同步当天所需数据，然后再输出评分
- `--sync-only` 表示只同步 `AKShare/Bundle -> SQLite`，同步完直接退出
- 首次同步会从 `2020-01-01` 全量建库；后续默认按“上次成功日期往前回溯 10 个交易日 -> 今天”做增量同步
- 一般不需要显式传 `--start-date`，直接用默认增量范围即可；只有你想强制缩小或重拉某一段历史时再手动指定
- 首次同步时间会比较长，尤其是成分股股息率部分
- 如果同一个 source 的缓存已经完整覆盖请求区间，再次执行会命中 checkpoint 并快速跳过
- 当 `akshare` 缺失旧版 `stock_a_indicator_*` 接口时，会自动退化为 `stock_value_em + stock_fhps_detail_em` 重建历史股息率
- `stock_zh_index_hist_csindex` 无法完整覆盖最新阶段的指数 PE，当前实现会额外叠加 `stock_zh_index_value_csindex`；中间若仍有空窗，评分时会自动降级为“不参与”

## 4. 第二步：直接看打分结果

如果你只是想看“当前最新可用日期”的评分，直接运行：

```bash
uv run python -m skyeye.dividend_scorer.main
```

输出 JSON：

```bash
uv run python -m skyeye.dividend_scorer.main --json
```

如果你想显式查看某一天，可以再传 `--date`。例如查看“今天”：

```bash
uv run python -m skyeye.dividend_scorer.main --date "$(date +%F)"
uv run python -m skyeye.dividend_scorer.main --date "$(date +%F)" --json
```

注意：

- 不指定 `--date` 时，命令会先自动检查是否需要同步，再取缓存里的最新可用日期
- 指定“今天”时，如果当天还没同步到缓存，或当天不是交易日，结果可能为空
- 所以“看最新结果”通常更推荐直接省略 `--date`

常用返回字段：

- `total_score`
- `score_percentile`
- `confidence`
- `features`
- `confidence_modifiers`
- `model_meta`
- `data_freshness`

## 5. 第三步：在策略中使用

### 5.1 运行滚动分位示例策略

推荐先用命令行临时开启 Mod，不要急着改全局配置：

```bash
uv run rqalpha run \
  -f skyeye/examples/dividend_low_vol_score_strategy_history_aware.py \
  -s 2024-01-01 \
  -e 2025-12-31 \
  --account stock 100000 \
  -fq 1d \
  -d ~/.rqalpha/bundle \
  -bm 000300.XSHG \
  -mc dividend_scorer.enabled true \
  -mc dividend_scorer.db_path ~/.rqalpha/dividend_scorer/cache.db
```

这一步主要是先确认三件事：

- 策略能正常跑完
- `get_dividend_score()` 能被策略读到
- 日志里的仓位变化方向符合预期

### 5.2 用 RQAlpha 原生回测跑单区间并生成图片

如果你已经开始调仓位曲线、热度控制或置信度逻辑，推荐再补一条单区间原生回测：

```bash
env UV_CACHE_DIR=/tmp/uv-cache MPLCONFIGDIR=/tmp/mplconfig uv run rqalpha run \
  -f skyeye/examples/dividend_low_vol_score_strategy_history_aware.py \
  -s 2024-05-03 \
  -e 2025-04-20 \
  --account stock 100000 \
  -fq 1d \
  -d ~/.rqalpha/bundle \
  -bm 512890.XSHG \
  --plot-save dividend_price_2405_2504.png \
  -mc dividend_scorer.enabled true \
  -mc dividend_scorer.db_path ~/.rqalpha/dividend_scorer/cache.db
```

说明：

- 这里把 benchmark 设成 `512890.XSHG`，就是为了直接和同一只 ETF 的买入持有对比
- `--plot-save dividend_price_2405_2504.png` 会把 RQAlpha 自带收益图直接保存到当前目录
- 这条命令更适合回答“这段时间策略到底有没有跑赢红利低波本身”
- 如果你在本地桌面环境直接跑，也可以去掉前面的 `UV_CACHE_DIR` 和 `MPLCONFIGDIR`

### 5.3 在你自己的策略里调用

只要启用 `dividend_scorer` mod，策略里就可以直接调用：

```python
from rqalpha.apis import *


def handle_bar(context, bar_dict):
    score = get_dividend_score()
    if not score or score.get("error"):
        return

    confidence_multiplier = {
        "normal": 1.0,
        "lowered": 0.5,
        "low": 0.0,
    }.get(score["confidence"], 0.0)

    if score["score_percentile"] <= 0.2:
        context.signal_target_percent = 0.95
    elif score["score_percentile"] >= 0.8:
        context.signal_target_percent = 0.0

    order_target_percent("512890.XSHG", context.signal_target_percent * confidence_multiplier)
```

策略侧会静默完成需要的数据准备，不会打印 CLI 那套同步进度日志。

这段代码只是“如何在你自己的策略里调用评分”的最小示例，不等同于仓库里的
`history_aware` 示例策略。

当前示例策略的交易逻辑：

- 每周最多调仓一次，避免日频噪声把仓位来回打碎
- 主信号不是固定 `20% / 80%` 二元开平仓，而是从评分历史里取最近 `5/15` 个有效 `score_percentile` 做快慢均值，再合成为 `p_ref = 0.7 * p_fast + 0.3 * p_slow`
- 主仓位由连续函数映射到目标仓位，整体大致在 `25% ~ 90%`，中性区故意做平，减少“刚加不久又减”
- `deadband + step_limit` 会限制单次调仓幅度；首次建仓允许更快，后续仓位调整更平滑
- 只有价格 / 溢价 / RSI 明显过热时，才会触发 heat override，把仓位临时压到更低，并进入短 cooldown
- 高位回补还要经过 `high reentry guard` 确认，不会一看到一点回落就立刻追补
- `confidence=low` 时不新增仓位；`confidence=lowered` 时主要体现为“放慢调仓速度”，而不是立刻清仓
- 打分结果里的 `buy_percentile_threshold / sell_percentile_threshold` 主要用于打分器审计和对照，不是当前 history-aware 策略的直接开平仓阈值

日频回测里要注意一件事：

- `get_dividend_score()` 现在返回的是“上一交易日已经确认”的分数
- `score["date"]` 是信号对应日期
- `score["trade_date"]` 是当前执行交易的日期

这样做是为了避免“用今天收盘后才能知道的分数，去成交今天收盘价”的前视偏差。

## 6. 第四步：策略研发与验收

### 6.1 参数审计与过拟合检查

如果你改了阈值、窗口或权重，先不要急着看收益，先跑参数审计：

```bash
uv run python -m skyeye.dividend_scorer.parameter_audit \
  --db-path ~/.rqalpha/dividend_scorer/cache.db
```

这份脚本会统一检查：

- `20 / 60 / 120` 日三个周期里，低分组是否持续优于高分组
- `2019-2021 / 2022-2023 / 2024-2026` 三段是否都通过
- 每段低分组、高分组样本是否足够，避免少量样本“看起来很准”
- 最终给出 `accurate / mixed / overfit_risk` 结论

如果你想把当前版本存成 baseline：

```bash
uv run python -m skyeye.dividend_scorer.parameter_audit \
  --db-path ~/.rqalpha/dividend_scorer/cache.db \
  --write-json /tmp/dividend_scorer_audit_baseline.json
```

以后新版本可以直接和旧 baseline 比较。当前脚本会额外检查 `20 / 60 / 120` 日整体 low-high spread 是否退化：

```bash
uv run python -m skyeye.dividend_scorer.parameter_audit \
  --db-path ~/.rqalpha/dividend_scorer/cache.db \
  --baseline-json /tmp/dividend_scorer_audit_baseline.json
```

### 6.2 用 `strategy_scorer` 做滚动窗口回测打分

如果你还想把交易层面的回测也一起验掉，可以直接跑：

```bash
uv run python skyeye/strategy_scorer.py \
  skyeye/examples/dividend_low_vol_score_strategy_history_aware.py \
  --mod dividend_scorer \
  -mc dividend_scorer.db_path ~/.rqalpha/dividend_scorer/cache.db \
  --log high
```

这条路径更适合：

- 看 `37` 个滚动窗口里的综合分、稳定分和市场环境适应性
- 检查策略是不是只在最近一个窗口好看，换个年份就失效
- 在改完参数后，快速比较“新版本 vs 旧版本”是否真的更稳

### 6.3 两条研发路径怎么配合

推荐顺序：

1. 先用 `rqalpha run --plot-save` 看你关心的单一区间，确认净值曲线、基准对比和仓位行为是否符合直觉。
2. 再用 `skyeye/strategy_scorer.py` 看滚动窗口得分，确认改动不是只对一个时间段有效。
3. 最后再回到参数审计，检查底层分数本身有没有退化。

## 7. 用诊断工具画出 `512890` 真实走势 vs 打分曲线

如果你现在主要想验证：

- ETF 处于相对低位时，打分是否也偏低
- ETF 处于相对高位时，打分是否也偏高

直接运行：

```bash
uv run rqalpha run \
  -f skyeye/tools/dividend_score_vs_price_plot.py \
  -s 2019-01-18 \
  -e 2026-03-06 \
  --account stock 100000 \
  -fq 1d \
  -d ~/.rqalpha/bundle \
  -bm null \
  --plot-save /tmp/dividend_score_vs_price.png \
  -mc dividend_scorer.enabled true \
  -mc dividend_scorer.db_path ~/.rqalpha/dividend_scorer/cache.db
```

说明：

- 这不是交易策略，而是诊断工具，不下单，只负责画图
- `-bm null` 是故意的，这样不会额外要求 benchmark 也覆盖整段区间
- `dividend_score` 就是打分器输出的 `total_score`
- `512890_close_norm` 不是实际价格，而是把 ETF 的收盘价映射到 `[0, 10]` 后得到的比较曲线
- `512890_close_norm` 使用的是“固定到回测结束日的前复权收盘价”，然后在整个回测区间做 min-max 归一化
- 两条线会出现在最下面的自定义 subplot 里；上半部分仍然是 RQAlpha 默认收益面板
- 如果你需要图形窗口，把 `--plot-save` 改成 `--plot`

建议重点观察：

- 两条线的阶段性顶部、底部是否大致同向
- 明显风格切换阶段，比如 `2021`、`2024`，打分拐点是否跟上
- 如果价格创新高而打分没有同步抬升，通常说明估值口径或底层数据还值得继续检查

说明一下日期区间：

- 上面的 `2019-01-18 -> 2026-03-06` 是我本地 bundle 上 `512890.XSHG` 实际验证通过的可用区间
- 如果你的 bundle 更完整，可以按你本地实际可用区间调整 `-s` 和 `-e`

## 10. 常见问题

### 10.1 报错 `akshare is required for sync_all`

原因：

- 你执行了 `--sync`
- 但当前环境没有安装 `akshare`

处理：

```bash
uv sync --python 3.13 --extra dividend-scorer
```

### 10.2 走代理时报 `ProxyError`

原因：

- 某些代理环境下，`Eastmoney` 链路会被中途断开
- 常见表现是 `fund_etf_hist_em` 或 `stock_value_em` 报 `ProxyError`

处理：

- 优先直接跑，不要给同步命令套代理
- 如果你开了类似 `fanqiang` 的环境变量，先取消再执行：

```bash
unset http_proxy https_proxy no_proxy
uv run python -m skyeye.dividend_scorer.main \
  --sync \
  --sync-only \
  --end-date "$(date +%F)"
```

### 10.3 报错 `etf_daily cache is empty, run sync_all first`

原因：

- SQLite 缓存还没建立
- 或 `db_path` 指错了

处理：

1. 先执行一次 `--sync`
2. 确认策略和 CLI 用的是同一个 `db_path`

### 10.4 打分器在策略里返回 `error`

常见原因：

- SQLite 数据缺列或缺日期
- 交易日覆盖率不够
- 股息率维度不可用
- 可用维度不足 3 个

建议先单独跑：

```bash
uv run python -m skyeye.dividend_scorer.main --json
```

先确认评分本身能正常算出来，再进策略。

### 10.5 画图时报 benchmark 或日期覆盖错误

常见原因：

- 你把 benchmark 设成了一个本地 bundle 覆盖不完整的标的
- 或者 `512890.XSHG` 在你的 bundle 里起止日期比命令里更短

处理：

- 对比图策略优先用 `-bm null`
- 根据你本地 bundle 的实际覆盖范围调整 `-s` 和 `-e`

### 10.6 `strategy_scorer` 里 2019 初期窗口分数很低

常见原因：

- `512890.XSHG` 上市日是 `2019-01-18`
- 但 `skyeye/strategy_scorer.py` 的默认滚动窗口仍从 `2016-02-01` 开始
- 所以前几个窗口虽然“跑通了”，但真正有持仓的交易日极少，年化收益/夏普/Sortino 会被稀疏样本放大

建议：

```bash
uv run python skyeye/strategy_scorer.py \
  skyeye/examples/dividend_low_vol_score_strategy_history_aware.py \
  --mod dividend_scorer \
  -mc dividend_scorer.db_path ~/.rqalpha/dividend_scorer/cache.db \
  --log high
```

当前版本会在输出里显式标记：

- `首次建仓滞后 X 天`
- `持仓仅 X/Y 个交易日`
- `窗口内无持仓日`

这些窗口更适合单独看，不适合直接混进综合分。

### 10.7 `strategy_scorer` 里卖出胜率 100%，是不是有问题

先分清楚这两个口径：

- `卖出胜率` 只统计“已经平仓的卖出”
- 它不包含期末仍然持有仓位的浮盈浮亏

另外滚动窗口本身是重叠的，所以同一个真实卖点会在多个窗口里重复出现。当前版本会额外提示：

- `卖出 N 笔只对应 M 个实际卖点日`

所以看到 `100%` 不代表策略没有回撤，更不代表没有过拟合；要结合：

- 窗口总收益率
- 期末是否仍有持仓
- 浮盈/浮亏
- 综合分与稳定性分

一起看。
